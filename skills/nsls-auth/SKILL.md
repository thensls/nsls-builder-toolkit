# NSLS Auth Integration

## SAFETY: THREE-TIER PERMISSION MODEL

This skill spans three permission tiers. Honor them.

1. **Read-only** — read the integration guide, the discovery document
   (`https://auth.nsls.org/.well-known/openid-configuration`), the JWKS,
   the app's existing auth code, and OIDC library docs. Free to do.
2. **Configuration** — write callback routes, env vars, identity columns,
   session middleware, logout routes inside the builder's app. Ask before
   touching files outside the obvious scope; explain what changes and where.
3. **Write to NSLS Auth** — registering a new OIDC client (Admin Portal UI
   or `npm run admin -- clients upsert`) changes shared infrastructure.
   Never proactively register. Produce a client-definition JSON, hand it to
   Kevin or a People-Ops admin, and let them run the registration. The
   `client_secret` must be generated by whoever runs the registration, not
   by Claude or the builder's chat.

## Purpose

Make centralized SSO trivial to wire into any NSLS app. NSLS Auth is the
single OIDC provider at `auth.nsls.org` for every NSLS-controlled product.
Without this skill, every builder rediscovers OIDC, mishandles PKCE or
state, or worst — keys identity off email and creates merge headaches when
emails change. With it, a builder gets the right library recommendation,
the right client registration, the right callback flow, and the right
session model in one pass.

## When To Use

- Adding "Sign in with NSLS" to a new app
- Migrating an existing app off its own password store
- Building a Society-branded sign-in (variant `societyLogin`)
- Wiring a React Native app to NSLS Auth
- Adding `admin` / `staff` role checks to an internal app
- Debugging a callback that returns `state` mismatch, `invalid_grant`,
  or 401 on `/oidc/userinfo`

## When Not To Use

- Adding auth that is **not** NSLS-managed (e.g. a third-party app you're
  integrating into NSLS infra). Use the third party's docs directly.
- App-specific role/permission systems that live entirely inside the app.
  NSLS Auth owns identity and coarse roles, not your app's authorization.

## The Macro: How NSLS Auth Works

NSLS Auth is the central OIDC provider. Every downstream app — Society
web, member portals, internal tools, mobile apps — delegates login to it.

Five facts to internalize before writing any code:

1. **SSO is redirect-based, not cookie-shared.** `app.nsls.org` and
   `thesociety.org` cannot share a browser cookie. SSO works because the
   browser bounces back through `auth.nsls.org`, which already has the
   user's auth-domain session.
2. **Two sessions, two owners.** NSLS Auth owns the auth-domain session.
   The app owns its own local session. After token exchange, create a
   local app cookie. Do not reuse NSLS Auth tokens as the app session.
3. **Authorization Code + PKCE always — no exceptions.** Not "PKCE for
   public clients only." Not "skip PKCE because we have a secret."
   Every authorization request gets a fresh PKCE verifier, fresh
   `state`, fresh `nonce`. A login path that produces fewer than three
   is broken. Treat as non-negotiable.
4. **`sub` is the durable identity key, not email — including in
   prototypes.** Email changes. The first user who updates their email
   creates a duplicate account or a silent identity switch. Persist
   `(auth_issuer, auth_subject)` on first login. After that, never look
   up users by email. "We'll fix it later" is how identity bugs ship.
5. **Pure SPAs are not supported.** Apps without a backend must add a
   backend-for-frontend (BFF) before integrating. Native mobile apps
   are supported as public clients.

## Decision Tree

Walk this before writing code or proposing a client registration.

### Web app or native?

| Your app | `application_type` | Auth method | Client secret | BFF host |
|---|---|---|---|---|
| Server-rendered (Rails, Django, Next.js with API routes) | `web` | `client_secret_post` or `client_secret_basic` | Required, server-only | App's own server |
| SPA + BFF (Next.js App Router with server actions, separate backend) | `web` | `client_secret_post` | Required, server-only | Vercel / Railway / Cloud Run |
| Static site on Netlify (presentations, dashboards) | `web` | `client_secret_post` | Required, set as Netlify env var | Netlify Edge Functions — see `references/netlify-app-implementation.md` |
| SPA, no backend | **STOP** | Add a BFF first | n/a | — |
| React Native / Expo / iOS / Android | `native` | `none` | Omitted | n/a |

`application_type` is **immutable** after registration. To switch web↔native, the client must be deleted and recreated.

### Branding variant?

| Variant | When to use | Notes |
|---|---|---|
| Generic (no `auth_page`) | Internal admin tools, default fallback | Uses `logo_uri` + `auth_ui` colors only |
| `societyLogin` | Society-branded apps (`thesociety.org`) | Hero image + bold intro copy + Society mark |
| `symfonyLogin` | Symfony-style sign-in pages | Sign-in only; other pages use generic shell |

### Scopes — request only what you need

| Scope | Request when | Default? |
|---|---|---|
| `openid` | Always | Required |
| `email` | App needs to display or store email | Almost always |
| `profile` | App needs name or picture | Usually |
| `offline_access` | App needs refresh tokens (long sessions without re-auth) | Only if needed |
| `roles` | App makes `admin` / `staff` decisions | Only if needed |

Do not request scopes the app doesn't use. Adding `roles` later is a client-config change, not a code rewrite.

### Trusted picture writes — default off

`can_manage_own_picture: true` is off by default and stays off. Set it to true only when the app has a documented need to update the user's canonical cross-app avatar (typically: account-management UI in a first-party app like Society). Reading `picture` does not require this flag. When unsure, leave it false — adding it later is a one-line client-config change.

## Quick Start: Server-Rendered Web App (Node)

The fastest happy path. See `references/web-app-implementation.md` for full code.

1. **Pick the library.** Use `openid-client` (panva) for Node. It handles
   discovery, PKCE, JWKS rotation, and ID token validation correctly.
2. **Add identity columns to the user model.**
   - `auth_issuer` (text)
   - `auth_subject` (text)
   - unique index on `(auth_issuer, auth_subject)`
   - claim snapshots: `email`, `email_verified`, `given_name`,
     `family_name`, `display_name`, `picture_url`
   - `last_authenticated_at`, `last_claim_sync_at`
3. **Set env vars.** Copy `templates/env.template` and fill in values.
4. **Draft the client-registration JSON.** Copy
   `templates/web-client-registration.json` and customize. Hand the JSON
   to Kevin or a People-Ops admin to register through the Admin Portal —
   do not register the client yourself.

   **About the `client_secret`.** Generate the secret locally using
   `openssl rand -hex 32` and immediately copy the output to a secure
   vault entry (1Password, Bitwarden). Share the secret with Kevin or the
   People-Ops admin who will register the client through the Admin Portal
   via that vault entry — not Slack, not email, not chat. The admin will
   paste the secret into the client registration form. Do not paste the
   secret into this chat or commit it anywhere; do not run
   `netlify env:list --plain` (or any equivalent that prints values to
   stdout) since that puts the secret into your terminal scrollback and
   any session transcript.
5. **Implement the routes.**
   - `GET /auth/login` — start authorization with new `state`,
     `nonce`, PKCE verifier; store all three server-side; redirect to
     `/oidc/authorize`.
   - `GET /auth/callback` — verify `state`, exchange code at
     `/oidc/token`, **fully validate ID token (signature against JWKS,
     `iss`, `aud`, `nonce`, `exp`) — every field, every time, even when
     the library claims to handle it**, find-or-create local user by
     `(issuer, sub)`, create local session cookie, redirect to app.
   - `POST /auth/logout` — clear local session, then redirect to
     `/oidc/logout?client_id=...&post_logout_redirect_uri=...`.
6. **Validate behaviors.** Use the checklist in the spec — first-time
   login, repeat login (SSO), userinfo claims, role claim shape,
   refresh failure, logout cleanup.

## Native App (React Native / Expo)

See `references/native-app-implementation.md` for full code.

Key differences from web:

- `application_type: "native"`, `token_endpoint_auth_method: "none"`,
  no client secret.
- Redirect URI uses a custom scheme like `thesocietyapp://oauth2/callback`.
- Use `expo-auth-session` (Expo) or `react-native-app-auth` (bare RN).
  Both handle PKCE and discovery correctly.
- Tokens land inside the app, not on a server. Store refresh tokens in
  the OS secure store (Keychain / Keystore), never in `AsyncStorage`.

## Migrating An Existing App

Order matters. Don't try to flip everything at once.

1. **Inventory** the current auth surface: login entry points, logout
   flow, callback URLs, user identifiers, local password storage,
   background jobs that depend on local sessions.
2. **Add identity columns** (`auth_issuer`, `auth_subject`, claim
   snapshots) without removing anything yet.
3. **Register the OIDC client** through the Admin Portal.
4. **Implement the new login + callback** behind a feature flag.
5. **Map existing users** on first login. Prefer linking on a verified
   business rule. Email-based linking is allowed only during the
   one-time migration window, and only once per user — the moment an
   email matches, persist `(issuer, sub)` and stop using email as a
   lookup forever after. When account collision is plausible, require
   explicit user confirmation rather than auto-linking.
6. **Switch the UI** to "Sign in with NSLS". Remove local password,
   registration, and reset flows from the app.
7. **Disable legacy credential paths** after cutover. Keep the
   profile-snapshot columns; drop the password column.

## Diagnostic Loop

When the integration breaks, the loop is **TRY → OBSERVE → DIAGNOSE → ADAPT → TRY AGAIN**. The most common failure modes:

| Symptom | First thing to check |
|---|---|
| `state` mismatch on callback | App lost server-side state between login and callback (different process, lost session, cleared store). Check `state` storage. |
| `invalid_grant` on token exchange | `redirect_uri` does not exactly match the one sent in the authorize request, or the code was already used, or PKCE verifier doesn't match. |
| ID token signature fails | JWKS cache stale, wrong `iss`, or local clock skew. Re-fetch JWKS and recheck `iss` against the issuer URL. |
| Login loop after callback | Session cookie not being set (SameSite, Secure, domain), or local session lookup fails immediately on the next request. |
| 401 on `/oidc/userinfo` | Access token expired or revoked. Clear local session and start a fresh login. |
| `roles` claim missing | App didn't request `roles` scope, OR the user has no active auth-wide roles. Both are valid; empty role set means "no auth-wide roles." |
| Picture write returns `503 temporarily_unavailable` | NSLS Auth has no `PROFILE_PICTURE_ALLOWED_HOSTS` configured for this environment. Talk to whoever runs the auth service. |
| Picture write returns `403 access_denied` | Client wasn't registered with `can_manage_own_picture: true`. Update the client config. |
| Logout doesn't return to branded login | `post_logout_redirect_uri` not registered exactly, or pointing to a placeholder dead-end. Register it as the app's login-start route, not a farewell page. |
| Refresh fails with `invalid_grant` | Central session was revoked (password reset, user disabled, role change in some cases). Delete stored refresh token, clear app session, force fresh login. |

For deeper diagnostics see `references/troubleshooting.md`.

## Common Mistakes

These come straight from the spec; every one of them has burned someone:

1. Embedding `client_secret` in browser JavaScript.
2. Skipping PKCE because "the client is confidential."
3. Using email as the durable user key.
4. Assuming a cookie can be shared between `app.nsls.org` and `thesociety.org`.
5. Treating an NSLS Auth access token as the app's session cookie.
6. Building app-local password reset, registration, or social login UI
   after migration.
7. Storing refresh tokens in browser JavaScript, `localStorage`, or
   `AsyncStorage` (React Native). Use server-side storage for web apps
   and OS secure storage (Keychain / Keystore) for native.
8. Treating `acr` / `amr` as authorization roles. They are auth context.
   Authorize off `roles[*].value`.
9. Assuming an already-issued ID token reflects a role change. Use
   `/oidc/userinfo` or force a fresh login when you need current roles.
10. Re-linking users by email on every login. Persist `(issuer, sub)` once.
11. Sending `post_logout_redirect_uri` that isn't registered exactly on
    the client.
12. Storing expiring signed image URLs as the canonical `picture`.

## Implementation Checklist

Use this in the PR description. Integration is complete when every box is true:

- [ ] App registered as an OIDC client in NSLS Auth
- [ ] App reads endpoints from discovery, not hardcoded
- [ ] Login uses authorization code + PKCE
- [ ] Callback verifies `state`
- [ ] Token exchange is server-side only (or in-app for native)
- [ ] ID token fully validated (`iss`, `aud`, `nonce`, `exp`, signature)
- [ ] App persists `(auth_issuer, auth_subject)` as unique identity key
- [ ] App creates its own local session
- [ ] App can fetch fresh claims via `/oidc/userinfo`
- [ ] If `offline_access` is requested: refresh tokens stored server-side
      (or in OS secure store for native), `invalid_grant` triggers full
      re-login, rotation persists new tokens
- [ ] If `offline_access` is NOT requested: confirmed app does not need
      long-lived sessions; this is a deliberate choice
- [ ] Local session cleared on logout
- [ ] If using RP-initiated logout, `post_logout_redirect_uri` registered
- [ ] If branded re-entry: post-logout URI returns to login-start, not a placeholder
- [ ] No client secret or refresh token in browser code
- [ ] Local password login removed (if migrating)

## Rationalizations You Will Have

Under pressure ("just ship the prototype", "this is internal", "the user is already authenticated"), agents drift. Every excuse below has a one-line counter. Catching one of these thoughts means stop.

| Excuse | Reality |
|---|---|
| "Email is fine as the key, this is a prototype." | Email changes. The first user who updates email creates a duplicate account. Use `(issuer, sub)`. Always. |
| "PKCE is for public clients; this is confidential." | The spec requires PKCE on every authorization request. No exceptions for confidential clients. |
| "I'll register the client myself, faster than asking." | Tier 3 operation. Hand the JSON to Kevin or a People-Ops admin. Never run `npm run admin -- clients upsert` or use the Admin Portal directly. |
| "I'll generate the `client_secret` here so we can paste it into the registration." | The admin running the registration generates the secret. Putting it in the chat puts it in the transcript. Leave the field as a placeholder. |
| "I'll embed the secret in the Next.js component, it's behind auth anyway." | Browser code is browser code. `client_secret` lives only in server runtime, never in any file shipped to the browser. No exceptions for "internal" or "behind auth." |
| "Token exchange succeeded, the ID token must be valid." | Token exchange success ≠ ID token validity. Validate signature, `iss`, `aud`, `nonce`, `exp` separately. Skipping this is the difference between OIDC and "I trust whoever sent me a JWT." |
| "I'll use `acr=password` to gate admin features." | `acr` and `amr` are authentication context, not authorization. Authorize off `roles[*].value`. Never off `acr` or `amr`. |
| "JWKS rotates rarely, I'll cache at startup." | Rotation breaks every cached-at-startup app the day it happens. Use a library that fetches on `kid` miss (`openid-client`, `jose` with `createRemoteJWKSet`, `authlib`). Never pin keys at boot. |
| "Relinking by email each login is simpler than persisting `sub`." | Persist `(issuer, sub)` on first login. Never look up by email after that. Email-on-every-login means an email change silently creates a new account. |
| "We'll add refresh handling later, ship without it." | Either request `offline_access` and handle refresh + `invalid_grant` correctly, or do not request `offline_access`. "Later" is how stale-token bugs ship. |
| "The library must validate the ID token, I don't need to check." | Some libraries do (`openid-client`, `react-native-app-auth`). Others don't (`expo-auth-session`). Verify what the specific library does and write the validation when it doesn't. |

## Red Flags — STOP

If any of these are true, stop and reconsider before continuing:

- The app is a pure browser SPA with no backend → add a BFF first.
- About to embed `client_secret` in client-side code → wrong tier; move
  to server.
- About to generate a `client_secret` to paste into the registration JSON
  → tier 3 violation; the admin generates it.
- About to use email as the unique user key → use `(issuer, sub)`.
- About to register a client without explicit Kevin / admin handoff →
  don't. Tier 3 operation.
- About to skip ID token validation because token exchange succeeded →
  don't. Validate signature, `iss`, `aud`, `nonce`, `exp`.
- About to skip ID token validation because "the library must handle it"
  → verify what the specific library does. `expo-auth-session` does NOT
  validate signatures. When unsure, write the validation.
- About to cache JWKS at startup with no rotation handling → use a
  library that fetches on `kid` miss; never pin keys.
- About to share a session cookie between `.nsls.org` and `.thesociety.org`
  → impossible; redirect through auth.nsls.org instead.
- About to copy NSLS Auth tokens into local storage → store server-side
  or in OS secure storage only.
- Tempted to use `acr` or `amr` for authorization → use `roles[*].value`.
